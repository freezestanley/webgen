## 后管系统方案模板

1. 页面类型 / 目标
   - 工具型后台页面，目标是让用户快速读状态、完成操作、减少误判。
2. 目标用户 / 使用场景
   - 高频使用的熟练用户，桌面优先，但 Pad 也需可操作。
3. 页面布局的壳
   - 本节是 **Discovery / Proposal 必填项**，属于后台工具系统页面的必做布局骨架，不能省略、不能降级为“后续补充”
   - 可折叠左侧多级导航
   - 固定顶部栏
   - 动态面包屑
   - 内部滚动主内容区
   - 移动端抽屉侧栏
   - 通知面板
   - 用户菜单
   - 示例路由占位页面
4. 设计拨盘
   - `DESIGN_VARIANCE = 4-6`
   - `MOTION_INTENSITY = 2-4`
   - `VISUAL_DENSITY = 6-8`
5. 配色 / 字体 / 图标
   - 中性灰阶统一，强调色只承担状态提示或主操作
   - 主界面不用 Serif
   - 数字密集区可搭配等宽数字
6. 素材 / 文案 / 接口
   - 优先读取真实字段与状态枚举
   - 接口未就绪时先定义 mock 数据结构
7. 技术实现
   - 延续现有组件系统和状态管理
   - 组件交互动画优先 Motion；无需为后台引入重滚动特效
   - 满屏布局优先固定页面壳体，并为主列表区、消息区、详情区定义独立滚动容器
   - 抽屉、弹窗、侧滑面板优先承担详情查看、补充编辑、辅助操作，不轻易打断主工作流
8. 状态设计
   - 列表 `loading / empty / error`
   - 操作按钮有 `pending / success / failure`
   - 过滤条件、分页、无权限、无结果要有反馈
   - 详情打开、抽屉展开、弹窗确认、栏位收缩都要有明确状态反馈与关闭路径
9. 响应式策略
   - Pad 竖屏优先改为分段堆叠，而不是硬保多栏
   - 触屏主操作区按钮热区不小于 `44 x 44px`
   - 横向表格必要时允许局部横滚，但主操作列应优先保留
   - 侧栏、筛选栏、会话栏在空间不足时应优先折叠或转抽屉，不强行压缩主内容区
10. 风险 / 待确认
   - 哪些字段是主决策信息
   - 哪些操作高风险，需要二次确认或撤销

## 登录 / 表单页方案参考

1. 页面类型 / 目标
   - 登录 / 注册 / 提交表单页，目标是低阻力完成输入与提交。
2. 目标用户 / 使用场景
   - 新用户或回访用户，常见于移动端单手操作与桌面快速录入。
3. 主要板块与信息层级
   - 标题与说明
   - 表单主体
   - 辅助说明 / 帮助入口
   - 次级操作：忘记密码、切换登录方式、返回
4. 设计拨盘
   - `DESIGN_VARIANCE = 3-5`
   - `MOTION_INTENSITY = 1-3`
   - `VISUAL_DENSITY = 4-5`
5. 配色 / 字体 / 图标
   - 保证对比度和可读性优先
   - 图标只辅助输入语义，不承担核心说明
6. 素材 / 文案 / 接口
   - 错误文案、校验规则、接口返回码需要尽量明确
7. 技术实现
   - 先保证原生可提交与校验反馈
   - 视觉动效只做轻反馈，不做重表演
8. 状态设计
   - 字段级错误
   - 提交中
   - 提交失败
   - 提交成功 / 下一步引导

## 后管系统规则

- 优先保证信息清晰、操作路径短、反馈及时。
- 减少无意义卡片包装。
- 高密度时依靠分隔、网格和留白组织信息。
- 动效只用于反馈状态、切换层级和局部引导。
- 高级感来自秩序、比例、统一细节，而不是炫光背景。
- 工具型页面采用满屏设计时，默认优先使用“页面壳体固定 + 内部业务区滚动”，不要把所有滚动都压给整页。
- 聊天窗口、消息列表、表格区、侧边详情区等高频内容区，默认应有明确的独立滚动容器。
- 聊天类界面默认要求：头部固定、输入区固定、消息区独立垂直滚动。
- 满屏后台默认避免无意双重滚动；若需要局部横滚或纵滚，应让滚动边界和责任区清晰可识别。
- 满屏壳布局默认允许栏目收缩，例如侧边导航、筛选栏、详情栏可折叠，以释放更多主操作空间。
- 工具型页面中的二级内容、详情内容、补充操作，默认优先考虑抽屉、弹窗、侧滑面板等按需展开容器，而不是直接挤占主工作区。
- 按钮触发的扩展内容应优先采用“就近展开、按需关闭、不中断主任务流”的交互方式，避免让用户在满屏工作台里频繁跳页丢上下文。
