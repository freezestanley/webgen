## 动效与特效技术选型规则

### Anime.js

适合：
- 轻量级页面动效
- 文案、按钮、卡片、区块入场
- Loading、hover、列表过渡、小型强调

优先条件：
- 目标是更灵动、更精致
- 不涉及复杂 3D
- 不涉及重滚动叙事
- 不需要复杂时间线控制

### Motion

适合：
- 组件级交互动画
- 页面状态切换
- layout transition
- React 场景中的现代 UI 动效

优先条件：
- 页面偏产品化
- 需要组件级状态反馈
- 需要更丝滑的声明式交互体验

### GSAP

适合：
- 滚动驱动动画
- 强控制力时间线
- 多元素联动演出
- 复杂叙事型 Hero

优先条件：
- 需要滚动叙事
- 需要复杂编排
- 需要高控制力

默认路由：
- 基础补间先看 `gsap-core`
- 时间线编排看 `gsap-timeline`
- 滚动驱动看 `gsap-scrolltrigger`
- React / Next 生命周期与 cleanup 看 `gsap-react`
- 收口优化看 `gsap-performance`

### Three.js

适合：
- 3D Hero 场景
- 粒子、空间、材质、光效
- 特效本身是核心记忆点的页面
- 背景元素特效，如：火星粒子燃烧飘动,logo或元素的粒子组成动画,流体动效,3D旋转特效等

优先条件：
- 特效是亮点而非点缀
- 有足够性能预算
- 能接受更高开发复杂度
- 已设计好降级策略

### React Bits（参考源，不默认直接接入）

定位：
- `React Bits` 更适合被视为**组件动效参考源 / 视觉效果参考源**，不是当前 `webgen` 模板体系里的默认运行时依赖。
- 当模板不是 React，或当前项目以原生 HTML / CSS / JS 为主时，可参考其效果样式与交互结构，再用现有技术栈复刻。

适合：
- 需要快速寻找更有设计感的按钮、卡片、文字显现、局部背景、hover、reveal、spotlight、magnetic 一类效果参考
- 需要为首页区块、展示模块、工具页重点操作区补一个更强记忆点
- 当前项目不是 React 模板，但希望借鉴成熟动效表现

不适用：
- 把 `React Bits` 当成当前原生模板的直接运行时依赖强塞进去
- 为了追求“更炫”复制整页重效果，破坏信息层级和性能预算
- 需要滚动叙事、复杂时间线、多段 pin / scrub 场景时

使用原则：
- 优先把 `React Bits` 当作“效果样例库”，拆解其：
  - 视觉节奏
  - 动效时序
  - hover / reveal / in-view 触发方式
  - 层次与材质表达
- 再用当前模板可接受的方案落地：
  - 简单入场、小交互：优先 CSS / `Anime.js`
  - 组件状态切换：React 项目才优先 `Motion`
  - 滚动驱动、多段编排：优先 `GSAP`
- 复刻时优先保留“效果意图”，不追求 1:1 照搬源码实现。
- 同一页面最多保留 1–2 个真正有记忆点的强效果，其余区域保持克制。
- 所有参考效果都必须经过：
  - 移动端表现检查
  - 文本可读性检查
  - 性能与降级检查
